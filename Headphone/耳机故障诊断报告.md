# 🎧 耳机插孔故障诊断报告

**设备**: 某 Intel Arrow Lake 平台笔记本  
**声卡**: HDA Intel PCH — Realtek ALC245  
**操作系统**: Linux (PipeWire 1.6.2 + WirePlumber)  
**诊断日期**: 2026-05-28

---

## 结论

**这是硬件原因。** 耳机插孔的物理检测机制失效，导致系统无法感知耳机已插入。

---

## 诊断过程

### 1. 硬件层检测 — 确定根因

使用 `hda-verb` 直接读取声卡编解码器寄存器：

```bash
sudo hda-verb /dev/snd/hwC0D0 0x21 GET_PIN_SENSE 0
# 返回值: 0x0 （未检测到插入）
```

正常插入耳机时应返回类似 `0x80000001` 的值（bit 31 = 存在标志）。`0x0` 说明插孔内部的**物理检测弹片**没有接触到耳机插头。

同时，输入设备节点也确认了该状态：

```
$ cat /sys/devices/.../input23/sw
# 文件为空 — 开关状态未被触发
```

| 检测项 | 结果 |
|--------|------|
| 耳机插孔 Pin Sense | `0x0` — 未检测到 |
| 预期正常值 | `0x800000xx` — 检测到插入 |
| 输入设备 SW 状态 | 未触发 |

### 2. 驱动层面 — 配置正确

编解码器节点 0x21（耳机引脚）配置完好：

| 参数 | 值 | 说明 |
|------|-----|------|
| Pin Widget Control | `0xc0` | OUT + HP 已使能 ✅ |
| EAPD | `0x2` | 已启用 ✅ |
| Unsolicited | `tag=01, enabled=1` | 中断检测已开启 ✅ |
| Connection Select | `0x1` (DAC 0x03) | 路由到耳机 DAC ✅ |

**驱动程序配置不是问题。** 系统已经正确设置了耳机引脚，只是物理检测信号没有到达。

### 3. 音频通路验证 — 耳机本身正常

| 测试场景 | 耳机出声？ |
|---------|-----------|
| **停止 PipeWire，ALSA 直出** (`aplay -D plughw:0,0`) | ✅ **有声音** |
| **禁用 ACP，PipeWire 正常播放** (`pw-play`) | ✅ **有声音** |

**耳机的音频通路是完好的。** 只要能正确配置声卡，耳机就能正常发声。

### 4. 排除的软件因素

| 怀疑点 | 排查结果 |
|--------|---------|
| PipeWire/WirePlumber 配置 | 已尝试 `api.acp.auto-port = true`，无效 |
| ACP 自动切换 | 禁用 ACP 后恢复正常（说明 ACP 本身没问题，是硬件检测信号缺失） |
| ALSA 混音器设置 | 手动 `amixer` 配置后通路易损 |
| 内核驱动参数 | `jackpoll_ms` 默认 0（使用中断检测），配置无异常 |
| WirePlumber 策略 | root cause 不在软件，在硬件检测信号 |

---

## 根本原因分析

```
插入耳机 → 插孔物理弹片应闭合 → 触发 GPIO 中断 → 声卡寄存器更新 → 系统感知
                ↓
          弹片未接触 ❌      ← 物理故障
                ↓
          Pin Sense = 0x0
                ↓
      系统认为"未插入耳机"
                ↓
      扬声器持续输出，耳机无信号
```

典型原因包括：
1. **插孔内部弹片氧化/变形** — 最常见
2. **插孔内异物（灰尘、棉絮）** — 导致接触不良
3. **耳机插头不兼容** — 某些插头的外径或触点位置有差异
4. **插孔焊接虚焊** — 主板级问题
5. **检测弹片断裂** — 需更换插孔

---

## 修复方案

### 方案一：清洁插孔（建议优先尝试）

用棉签蘸少量无水酒精或电子清洁剂擦拭插孔内部，反复插拔耳机几次，利用插头机械动作恢复弹片接触。

### 方案二：软件切换（已完成）

创建一键切换脚本 `switch-audio.sh`，配合 WirePlumber 配置禁用 ACP，通过 `amixer` 直接控制硬件切换扬声器/耳机输出。

**原理**：绕过失效的物理检测，由用户在软件层面手动切换音频输出通道。

### 方案三：维修

如清洁无效，需送修更换耳机插孔（3.5mm 音频插座）。

# 🎧 Headphone Jack Diagnostic Report (English)

**Device**: Laptop with Intel Arrow Lake platform  
**Audio Codec**: HDA Intel PCH — Realtek ALC245  
**OS**: Linux (PipeWire 1.6.2 + WirePlumber)  
**Date**: 2026-05-28

---

## Conclusion

**This is a hardware issue.** The physical jack detection mechanism inside the headphone socket has failed, causing the system to be unaware that headphones are plugged in.

---

## Diagnostic Process

### 1. Hardware-Level Detection — Root Cause

Used `hda-verb` to read the audio codec registers directly:

```bash
sudo hda-verb /dev/snd/hwC0D0 0x21 GET_PIN_SENSE 0
# Result: 0x0 (not detected)
```

A properly inserted headphone should return ~`0x80000001` (bit 31 = presence flag). `0x0` indicates the **physical detection switch inside the jack** is not making contact with the headphone plug.

The input device node also confirmed this:

```
$ cat /sys/devices/.../input23/sw
# File empty — switch state was never triggered
```

| Test | Result |
|------|--------|
| Headphone Jack Pin Sense | `0x0` — not detected |
| Expected normal value | `0x800000xx` — detected |
| Input device SW state | Not triggered |

### 2. Driver Configuration — Correct

Codec node 0x21 (headphone pin) is properly configured:

| Parameter | Value | Status |
|-----------|-------|--------|
| Pin Widget Control | `0xc0` | OUT + HP enabled ✅ |
| EAPD | `0x2` | Enabled ✅ |
| Unsolicited | `tag=01, enabled=1` | Interrupt detection on ✅ |
| Connection Select | `0x1` (DAC 0x03) | Routed to headphone DAC ✅ |

**The driver configuration is not the problem.** The system correctly sets up the headphone pin; the physical detection signal simply does not arrive.

### 3. Audio Path Verification — Headphones Work

| Test Scenario | Headphone Audio? |
|--------------|:----------------:|
| **Stop PipeWire, direct ALSA output** (`aplay -D plughw:0,0`) | ✅ **Yes** |
| **Disable ACP, normal PipeWire playback** (`pw-play`) | ✅ **Yes** |

**The headphone audio path is intact.** As long as the codec is properly configured, the headphones produce sound normally.

### 4. Eliminated Software Factors

| Suspect | Investigation Result |
|---------|---------------------|
| PipeWire/WirePlumber config | Tried `api.acp.auto-port = true` — no effect |
| ACP auto-switching | Works normally with ACP disabled (proves the issue is missing hardware detection signal, not ACP itself) |
| ALSA mixer settings | Audio path works with manual `amixer` configuration |
| Kernel driver parameters | `jackpoll_ms = 0` (unsolicited events), no abnormalities |
| WirePlumber policy | Not a software policy issue — root cause is at the physical layer |

---

## Root Cause Analysis

```
Plug in headphones → Jack's physical switch should close → GPIO interrupt → Codec register updates → System detects
                           ↓
                    Switch not contacting ❌      ← Physical failure
                           ↓
                    Pin Sense = 0x0
                           ↓
            System thinks "no headphones plugged in"
                           ↓
            Speakers stay active, no audio to headphones
```

Typical causes:
1. **Oxidized or deformed contact leaf inside the jack** — most common
2. **Foreign debris (dust, lint) inside the jack**
3. **Incompatible headphone plug** — diameter or contact position variation
4. **Cold solder joint on the motherboard**
5. **Broken detection leaf** — requires jack replacement

---

## Solutions

### Option 1: Clean the Jack (Recommended First)

Use a cotton swab dipped in isopropyl alcohol or electronic contact cleaner. Insert and remove the headphone plug several times to mechanically restore contact.

### Option 2: Software Toggle (Implemented)

Created a one-click toggle script `switch-audio.sh` with a WirePlumber configuration that disables ACP. Uses `amixer` to directly control the hardware for switching between speaker and headphone output.

**Principle**: Bypasses the failed physical detection by allowing the user to manually switch audio output channels in software.

### Option 3: Hardware Repair

If cleaning does not help, the 3.5mm audio jack needs to be replaced (requires disassembly and soldering).
