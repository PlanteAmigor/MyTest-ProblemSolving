## Analysis & Root Cause

### 1. Why NaN appears with long durations

NaN in diffusion models for long audio is a **latent-space drift** problem, not an out-of-memory issue (you have 128 GB RAM / ~115 GB shared VRAM):

| Factor | Explanation |
|--------|-------------|
| **Training distribution** | The DiT and VAE are trained on audio clips of limited duration. Beyond that range, the latent representations can drift outside the trained distribution. |
| **Attention instability** | Longer sequences produce larger attention logits, increasing the risk of softmax overflow and numerical divergence. |
| **Positional encoding saturation** | Positional embeddings may not generalize well to sequence lengths far beyond those seen during training. |
| **VAE decoder sensitivity** | Once latents drift outside the expected distribution, the VAE decoder can produce extreme values → NaN in the final waveform. |

### 2. Why Windows crashes harder than Linux (240s vs 600s)

| Platform | Speed | Driver aggressiveness | Thermal/power stress | NaN threshold |
|----------|-------|-----------------------|---------------------|---------------|
| Windows 11 | 11–12 tok/s | High (higher clocks) | High → crash/reboot at ~240s | Lower |
| Ubuntu 26.04 | 4–5 tok/s | Low (conservative) | Low → stable up to ~600s | Higher |

- The Windows driver pushes the iGPU to higher performance, which increases thermal/power stress. Under sustained load, this can trigger driver crashes, assertion failures, or even system reboots.
- On Linux, the same workload runs slower but cooler, so the system survives longer. The NaN still appears at 600s, but as a **purely numerical issue** without system-level crashes.
- **Important evidence**: On Ubuntu, after a NaN at 600s, the next 300s generation worked perfectly — this confirms the GPU hardware is fine; the NaN is a software/numerical issue.

### 3. Why "NaN cascade" on Windows but not on Linux

On Windows, NaN causes a **system-level crash** (assertion failure / reboot), which corrupts GPU driver state. Subsequent generations crash immediately.

On Linux, NaN is a **soft failure** — the tensor contains NaN but the GPU driver remains healthy. The next generation resets the computation graph and runs normally.

### 4. Suggested mitigations (configurable, no code changes needed)

| Mitigation | How to apply |
|------------|-------------|
| **Use shorter durations** | Stick to ≤ 360s (6 min) on Windows, ≤ 480s (8 min) on Linux |
| **Use `velocity_norm_threshold`** | If available in the UI, set to 2.0–3.0 to clamp outlier predictions during diffusion |
| **Enable INT8 quantization** | Reduces memory bandwidth pressure; may help with numerical stability |
| **Reduce thermal stress on Windows** | Use Intel Arc Control to lower power limit or GPU clock speed |
| **Wait for Intel Linux XPU driver updates** | Intel is actively improving Linux XPU performance; future drivers may close the speed gap |

### 5. Key takeaway

> **The NaN at long durations is a model limitation (latent drift), not a hardware defect.**
>
> Windows is faster but less stable for long generations due to aggressive driver behavior.
> Linux is slower but more reliable — you can safely generate up to ~480s (8 min) on Ubuntu.

## 分析 & 根本原因

### 1. 为什么长音频会出现 NaN

长音频 NaN 是**潜变量空间漂移**问题，不是显存不够（你有 128GB 内存/约 115GB 共享显存）：

| 因素 | 说明 |
|------|------|
| **训练分布限制** | DiT 和 VAE 训练时使用的音频时长有限，超出范围的潜变量会偏离训练分布 |
| **注意力不稳定** | 长序列产生更大的 attention logits，增加 softmax 溢出的风险 |
| **位置编码饱和** | 位置嵌入对远超训练时长的序列泛化能力有限 |
| **VAE 解码器敏感** | 潜变量漂移后，VAE 解码器可能输出极端值 → 最终波形出现 NaN |

### 2. 为什么 Windows 比 Linux 更容易崩溃（240s vs 600s）

| 平台 | 速度 | 驱动激进程度 | 热/功耗压力 | NaN 出现阈值 |
|------|------|------------|------------|-------------|
| Windows 11 | 11–12 tok/s | 高（更高频率） | 高 → ~240s 崩溃/重启 | 更低 |
| Ubuntu 26.04 | 4–5 tok/s | 低（保守调度） | 低 → 稳定到 ~600s | 更高 |

- Windows 驱动让 iGPU 跑在更高性能，持续负载下功耗/温度升高 → 触发驱动崩溃、断言失败甚至系统重启
- Linux 上同样负载跑得更慢但更凉快，所以系统能撑更久。NaN 在 600s 仍然出现，但只是**纯数值问题**，不会导致系统级崩溃
- **重要证据**: Ubuntu 上 600s NaN 后下一次 300s 生成完全正常 — 说明 GPU 硬件没问题，NaN 是软件/数值问题

### 3. 为什么 Windows 上出现 "NaN 级联"，Linux 不会

- **Windows**: NaN 导致**系统级崩溃**（断言失败/重启）→ GPU 驱动状态损坏 → 后续生成必定失败
- **Linux**: NaN 是**软故障** — 张量里有 NaN，但 GPU 驱动状态健康 → 下一次生成重置计算图，正常运行

### 4. 缓解方案（不改代码，仅配置）

| 方案 | 操作 |
|------|------|
| **控制时长** | Windows 上 ≤ 360s（6 分钟），Linux 上 ≤ 480s（8 分钟） |
| **开启 `velocity_norm_threshold`** | 如果 UI 中有此选项，设为 2.0–3.0，可在扩散过程中钳制异常预测值 |
| **开启 INT8 量化** | 减少内存带宽压力，有助于数值稳定 |
| **降低 Windows 热压力** | 用 Intel Arc Control 降低功耗限制或 GPU 频率 |
| **等待 Intel Linux XPU 驱动更新** | Intel 正在持续优化 Linux XPU 性能，未来驱动可能缩小速度差距 |

### 5. 核心结论

> **长音频 NaN 是模型本身的数值限制（潜变量漂移），不是硬件缺陷。**
>
> Windows 更快，但由于驱动激进，长音频稳定性更差。
> Linux 更慢但更可靠 — Ubuntu 上可以安全生成到约 480s（8 分钟）。
